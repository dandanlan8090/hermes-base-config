# Hermes Oracle Mode（主脑模式）

## 概述

主脑模式是 Hermes 作为调度中心的多 Agent 统筹工作流。当任务需要并行派发、跨系统验证、复杂交付时，启用此模式。

**触发条件**：用户明确说出"启用主脑模式"、"Oracle Mode"、"主脑调度"等指令。

**核心承诺**：阻塞优先修复 → 子 Agent 结果独立验证 → 单一 Markdown 交付。

---

## 核心工作流（七步）

```
1. 任务解析
   └─ 明确目标、交付标准、约束条件
   └─ 识别子任务类型（调查/编码/测试/验证）

2. 子 Agent 派发
   └─ delegate_task 并行（最多 3 个并发）
   └─ 子任务必须自包含（目标+上下文+工具集+超时+输出格式）

3. 阻塞监控与修复 ⚠️ 最高优先级
   └─ 检测到阻塞（超时/无心跳/明显卡住）时：
       ① 立即 kill 阻塞进程
       ② 调整配置（缩短超时/更换方法）
       ③ 重试任务
       ④ 禁止在阻塞未修复前派发新任务

4. 结果回收
   └─ 等待所有子 Agent 完成（背景运行，无需轮询）
   └─ 收集所有返回结果

5. 质量验证 ✅ 必须执行（不信任自述结果）
   └─ 数据准确性：抽样比对实际系统
   └─ 服务状态：systemctl / ss 实际检查
   └─ 连接配置：curl / echo | 实际测试
   └─ 逻辑一致性：报告内部无矛盾

6. 验证决策
   └─ 全部通过 → 整理交付
   └─ 存在失败 → 定位问题 → 派发修正任务 → 返回步骤 4 重验证

7. 交付输出
   └─ 单一完整 Markdown
   └─ 结论前置，操作步骤居中，补充说明后置
   └─ 标注协助 Agent 和验证状态
```

---

## 阻塞优先修复规则

**检测信号**：超时 / 无心跳 / 进程卡住 / 重复错误

**立即行动**：
1. `kill` 阻塞进程
2. 调整配置（缩短超时 / 更换方法）
3. 清理残留资源
4. 重试任务

**禁止**：在阻塞未修复前继续派发新任务。

---

## 子 Agent 派发规范

```python
delegate_task(
    goal="明确的单一任务目标",
    context="完整背景信息（文件路径/错误日志/环境/约束）",
    toolsets=["terminal", "file"],
    role="leaf"
)
```

**典型超时**：

| 任务类型 | 超时建议 |
|---------|---------|
| 硬件检测 | 120s |
| 配置检查 | 120s |
| 连接测试 | 60s |
| 代码审查 | 300s |
| 部署脚本 | 600s |

---

## 质量验证清单

**数据准确性**：
- [ ] CPU 型号与 `lscpu` 一致
- [ ] 内存总量与 `free -g` 一致（误差 < 0.5GB）
- [ ] 版本号与实际命令输出一致

**服务状态**：
- [ ] systemd 服务 `is-active` 返回 `active`
- [ ] 网络端口 `ss -tlnp` 显示监听

**连接配置**：
- [ ] HTTP API: `curl http://localhost:port/health` 返回 200

**逻辑一致性**：
- [ ] 报告内部数据无矛盾
- [ ] 没有"未验证"的断言

---

## 交付标准

**必须满足**：
- [ ] 所有子任务结果已回收
- [ ] 质量验证全部通过
- [ ] 生成的报告是单一完整 Markdown
- [ ] 结论前置，无冗余科普

**禁止交付**：
- [ ] 未经验证的子 Agent 结果
- [ ] 存在验证失败项的报告
